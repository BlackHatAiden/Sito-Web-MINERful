Declare Process Specification

Process Mining


Process mining aims at discovering, monitoring and improving real processes by extracting knowledge from event logs readily available in today's information systems as described by Wil Van Der Aalst. There are several process mining techniques but all of them have an event log as a starting point. Each technique assumes that it is possible to record events sequentially so that each of them relates to a particular activity and is associated with a particular process instance. A process instance is a single process execution. Logs can store additional information about events such as the resource (i.e. the person or device) executing Task, the timestamp of the event (i.e. an initial process value e.g. start time), or data elements recorded with the event.

Event logs can be used to conduct three types of process mining:

  • Process Discovery: The process discovery technique takes a log of events as input and outputs a model without using any a priori information. This is the most widely used and important technique in process mining.


  • Conformance Checking: The conformance checking technique takes as input an existing process model and compares it with an event log of the same process. Conformance checking can be used to check whether reality, as recorded in the log, conforms to the model and vice versa.


  • Enhancement: The enhancement technique exploits the idea of extending or improving an existing process model taken as input, using information about the actual process recorded in some event log. The main difference between conformance checking and enhancement is that in this technique the alignment between model and reality is measured, aiming to change or extend the model a prior.


Unlike existing analysis approaches, process mining is process-centred (not data-centred), truly intelligent (learning from historical data) and fact-based (relying on event data).

What is a declarative process?


Declarative process models consist of a set of constraints exerted over the execution of process activities. DECLARE is a declarative process modelling language that specifies a set of constraint templates along with their graphical notation. The automated discovery of DECLARE models aims at finding those constraints that are verified throughout a given event log.

There are three important focuses :

  • Open model


  • Specify constraints for permitted behaviour


  • Every execution that complies with them is acceptable.


The Declarative Process Miner is a process mining technique based on declarative process modelling languages. It takes a process model as input in addition to the event log. Furthermore The Declarative Process Miner take advantage of declarative process models like Declare, unlike traditional process mining techniques, they exploit Petri nets based on procedural modelling languages where all possible ordering of events must be made explicit.
An important feature of Declare is precisely its definition of relationships between activities through models of constraints. Each of these models has its own semantics described in LTL terms, a unique name and a graphical representation.

Declarative Model


Declarative Process Modeling is a process modelling approach that specifies the order and alternative ways of execution for a workflow in an implicit manner through the use of constraints. Furthere the Declarative Process Modeling allow modelers to capture constraints on the allowed activity flows, meaning that all flows are allowed provided that they do not violate the specified constraints.
(Open Model).

vs

Imperative Model


The classical approach is called “imperative” (or also “procedural”) because it explicitly represents every step allowed by the process model at hand, by means of transitions (the possible actions to do) among places/states (the legal situations where the process can wait or terminate). This leads to the likely increase of graphical objects as the process allows more alternative executions. The size of the model, though, has undesirable effects on understandability and likelihood of errors. Further Imperative Process Modeling is also defined like A process modelling approach that specifies the order and alternative ways of execution for a workflow in an explicit manner through the use of graphical constructs. In the end every flow that is not specified is implicitly disallowed
(Closed Model).

A declarative process specification in Declare




If an abstract is submitted, a new paper had been written or will be written RespondedExistence(Submit abstract, Write new paper)



After the paper submission, a confirmation email is sent Response(Submit papers, Send confirmation email)



Only after the paper submission, the paper will be reviewed Succession(Submit papers, Review paper)



A paper can be accepted only after it has been reviewed Precedence(Review paper, Accept paper)



After the rejection, no further submission follows NotSuccession(Reject paper, Submit paper)



Paper cannot be both accepted and rejected NotCoExistence(Accept paper, Reject paper)
